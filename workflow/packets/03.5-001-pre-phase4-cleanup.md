### F3-2 — Shared SQLite connection with WAL and checked writes

- [ ] A single shared `sqlite3` connection to `library.sqlite` is opened once in
      `TapeScrapeApp` and passed to both `SQLiteLibraryRepository` and
      `SQLitePlaybackHistoryRepository` (and later the download repository). The
      connection is owned by a new lightweight **`LibraryDatabase` actor** that:
  - Opens the connection with `PRAGMA journal_mode=WAL` and
    `PRAGMA busy_timeout=3000` (3 s — generous for single-user).
  - Exposes the `OpaquePointer` to repository actors via an init parameter (each
    repository no longer opens its own connection).
  - Owns `sqlite3_close` in its `deinit`.
- [ ] Every `sqlite3_step` call on a write statement (INSERT/UPDATE/DELETE) checks the
      return code. On `SQLITE_BUSY` or any non-`SQLITE_DONE` result, log a warning via

      no longer *silent*).
- [ ] `SQLiteLibraryRepository` and `SQLitePlaybackHistoryRepository` no longer call
      `sqlite3_open` or `sqlite3_close` themselves. They accept the `OpaquePointer` (or
      the `LibraryDatabase` actor) at init.
- [ ] The duplicated `private let SQLITE_TRANSIENT = …` stays in each file for now
      (F3-3 convention: DRY only on a 3rd impl), but schema creation (`CREATE TABLE IF
      NOT EXISTS`) is run once by `LibraryDatabase`, not independently by each repo.
- [ ] `TapeScrapeApp` constructs one `LibraryDatabase`, passes it to both repos.
- [ ] All existing Swift tests pass. No behavior change beyond the WAL/busy_timeout

## Interface sketch

```swift
// LibraryDatabase.swift — shared connection owner
actor LibraryDatabase {
    nonisolated(unsafe) private var db: OpaquePointer?

    init(url: URL) {
        var dbPointer: OpaquePointer?
        if sqlite3_open(url.path, &dbPointer) != SQLITE_OK {
            fatalError("[LibrarySQLite] cannot open \(url.path)")
        }
        db = dbPointer
        // WAL + busy_timeout for concurrent readers/writers
        sqlite3_exec(db, "PRAGMA journal_mode=WAL", nil, nil, nil)
        sqlite3_exec(db, "PRAGMA busy_timeout=3000", nil, nil, nil)
        Self.createLibraryTables(db!)
        Self.createHistoryTables(db!)
    }

    deinit { if let db { sqlite3_close(db) } }

    var pointer: OpaquePointer? { db }
}

// SQLiteLibraryRepository — no longer owns connection
actor SQLiteLibraryRepository: LibraryRepository {
    private let db: OpaquePointer?

    init(database: LibraryDatabase) async {
        self.db = await database.pointer
    }

    // Checked write helper
    private func checkedStep(_ stmt: OpaquePointer?, context: String) {
        let rc = sqlite3_step(stmt)
        if rc != SQLITE_DONE {
            print("[LibrarySQLite] \(context) failed: \(rc)")
        }
    }
    ...
}

// TapeScrapeApp — one database, two repos
init() {
    let database = LibraryDatabase(url: TapeScrapeApp.dbURL)
    let history = SQLitePlaybackHistoryRepository(database: database)
    let lib = SQLiteLibraryRepository(database: database)
    ...
}
```

## Known ambiguities / open questions

- **`LibraryDatabase` as actor vs plain class.** The sketch uses `actor` for consistency
  with the repos, but its only job is init/deinit + exposing the pointer. A plain class
  would also work. Prefer actor — it matches §18 convention and the pointer access is
  naturally serialized. If the async init of repos (needed to `await database.pointer`)
  creates ergonomic issues, a synchronous alternative (pass the `OpaquePointer` directly
  after awaiting it once in `TapeScrapeApp`) is acceptable.
- **Schema creation ownership.** Moving `CREATE TABLE` from each repo into
  `LibraryDatabase` means the database actor knows about both schemas. This is a
  pragmatic trade — the alternative (each repo creates its own tables on first use) is
  how it works today but means the database actor can't guarantee a consistent schema
  state. Prefer centralized creation.
