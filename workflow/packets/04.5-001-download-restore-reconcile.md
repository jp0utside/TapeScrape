## Goal

Close **F4-1**, the Phase-4 boundary review's only 🔴 blocker. The in-memory
`recordingProgress` mirror in `DownloadManager` desyncs from the authoritative
`DownloadRepository` during launch-time restore, so a fully-downloaded recording can
display as `.downloading`/`.failed`. Fix the restore path so the mirror is reconciled
*from* the repository for **all** states, and replace the test's `Task.sleep` race with a
deterministic awaitable restore hook. Done = the Swift suite is green
(`xcodebuild test` → `** TEST SUCCEEDED **`) with no regression to the `04-003`
resilience behavior.

## Root cause (from the Phase-4 review — do not re-derive)

`DownloadManager.init` fires `Task { await restoreState() }`
(`DownloadManager.swift:25`) with no ordering vs. later repo mutation. `restoreState()`
seeds `recordingProgress` from one `allDownloads()` read (`DownloadManager.swift:88-92`),
then `rehydrateTaskMap()` does a **second** `allDownloads()` (`:112-122`) and writes the
mirror **only** for active-task or still-`.downloading` records. A record that became
`.downloaded` or `.failed` between the two reads keeps the stale snapshot-1 value — the
mirror diverges from the repository (violates CONVENTIONS §20: "the repository is the
authority — the mirror must be reconciled *from* it"). The retry no-op itself is correct
(`failedTracks` empty → `guard` returns, `:59-60`); the defect is the un-reconciled
restore. `DownloadManagerTests.retryDownloadDoesNothingWhenAllTracksComplete()` fails
deterministically (3/3 in isolation) because its `Task.sleep(100ms)` races the init
restore Task.

## Interface sketch

```swift
@Observable @MainActor
final class DownloadManager: NSObject {
    private var restoreTask: Task<Void, Never>?

    init(storage: AudioStorage, repository: any DownloadRepository) {
        ...
        restoreTask = Task { await restoreState() }
    }

    /// Test/diagnostic hook — completes when launch-time restore has finished.
    /// Production callers never need to await this.
    func whenRestored() async { await restoreTask?.value }

    private func restoreState() async {
        await rehydrateTaskMap()          // build taskMap, cancel orphans,
                                          // mark interrupted downloading → failed in repo
        let activeIDs = Set(taskMap.values.map(\.identifier))
        let downloads = await repository.allDownloads()   // SINGLE authoritative read
        for record in downloads {
            if activeIDs.contains(record.identifier) {
                let p = Double(record.completedTracks) / Double(max(record.totalTracks, 1))
                recordingProgress[record.identifier] = .downloading(progress: p)
            } else {
                recordingProgress[record.identifier] = record.state   // verbatim
            }
        }
    }

    private func rehydrateTaskMap() async {
        // unchanged: getAllTasks → match via findTrackByStreamURL → cancel orphans;
        // mark any `.downloading` record with no active task `.failed` in the REPO.
        // (No longer seeds recordingProgress itself — restoreState does the single
        //  reconcile pass after this returns.)
    }
}

// DownloadManagerTests — deterministic, no Task.sleep for restore
let manager = DownloadManager(storage: storage, repository: repo)
await manager.whenRestored()
manager.retryDownload(identifier: "gd77-05-08.sbd")
// (retryDownload still dispatches an internal Task; a short wait there is acceptable
//  if no clean hook exists — but the RESTORE race must be removed via whenRestored())
```

## Known ambiguities / open questions

- **Hook shape: stored `Task` + `whenRestored()` vs a `private(set) var hasRestored`
  flag.** Prefer the stored-`Task`/`await` form — it is race-free and needs no polling.
  A bool flag would reintroduce a sleep/poll in tests. Either is acceptable if the test
  becomes deterministic; the stored-Task form is the recommended one.
- **`retryDownload` internal `Task`.** `retryDownload` dispatches its own `Task`
  (`DownloadManager.swift:58`); this packet only removes the *restore* race. If the
  post-retry assertion still needs a brief wait for that inner Task, that is acceptable
  for v1 (the `04-003` summary already noted it) — do **not** expand scope to redesign
  `retryDownload`'s concurrency here. Only the restore desync (F4-1) is in scope.
- **Ordering of rehydrate vs. mirror seed.** Rehydrate must run first so the
  interrupted→failed repo mutation is reflected in the single final `allDownloads()`
  read. Confirm this order; it is the crux of the fix.
